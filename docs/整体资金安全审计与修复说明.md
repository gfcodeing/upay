## 〇、部署约束（重要，务必遵守）

**本系统按「单实例部署」设计，禁止多实例/多进程/多机水平扩展。** 原因：

1. 下单流程用进程内 `sync.Mutex`（`web/function.go` 的 `sync_mu`）串行化，多实例时各进程的锁互不可见，金额分配会冲突。
2. 数据库为 SQLite（`db/sdb/sdb.go`），本身不支持多机共享写。
3. 金额占用、回调锁依赖单一 Redis 实例。

如未来确需多实例：必须把数据库换成 MySQL/PostgreSQL、金额唯一性完全交给共享 Redis 的 `SetNX`、并对 `(Token, ActualAmount, Status)` 建唯一约束，届时另行评估。

## 一、本次修复（优先级一：堵住重复入账 / 超额发货）

整体审计发现，所有「重复入账」类风险的总根源是：**入账动作不是原子、不防重**。具体表现为三处，已统一修复：

### 1. 新增统一原子入账函数 `sdb.MarkOrderPaid`
位置：`db/sdb/sdb.go`。所有入账路径**必须**调用它，禁止再用 `DB.Save(&order)` 裸覆盖。三重保障：

- **状态守卫**：`WHERE status = StatusWaitPay` 条件更新。已过期/已支付的订单不会被改动 → 杜绝「已过期订单被扫块复活成支付成功」。
- **txHash 去重**：`block_transaction_id` 配合数据库唯一索引，同一笔链上交易只能入账一个订单。
- **只更新受影响列**（`status` + `block_transaction_id`），不全字段覆盖 → 不会覆盖回调状态等并发写入的字段。

返回 `true` 仅当确实由「待支付」改为「支付成功」（`RowsAffected==1`），调用方据此决定是否触发回调。

### 2. `block_transaction_id` 部分唯一索引
位置：`db/sdb/sdb.go` 的 `Start()`，建表后执行：
```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_orders_block_tx_id
ON orders(block_transaction_id)
WHERE block_transaction_id <> '' AND block_transaction_id IS NOT NULL
```
用「部分索引」是因为大量待支付订单该字段为空串，普通唯一索引会因多个空串冲突。作为重复入账的最后防线。

### 3. 全部入账路径改走 `MarkOrderPaid`（共 14 处）
- 扫块/API 各币种（12 个文件）：`tron`、`trx`（含 Grid）、`ERC20_USDT`、`USDC_ERC20`、`USDC_Polygon`、`USDT_Polygon`、`USDT_ArbitrumOne`、`USDC_ArbitrumOne`、`BSC_USD`、`USDC_BSC`。
- BSC 扫块兜底：`BSC_USD/bsc_scanner.go`、`USDC_BSC/usdc_bsc_scanner.go`。
- **手动补单** `web/web.go` `/api/manual-complete-order`：
  - 新增状态校验：只有 `StatusWaitPay` 的订单可补单，已支付/已过期拒绝。
  - 改走 `MarkOrderPaid`，txHash 用 `"manual:订单号"` 标记（区分人工补单、满足唯一索引）。
  - 仅当返回 `true` 才触发回调 → 从根本上与自动扫块互斥，杜绝双重入账。

### 附带收益
- cron 对 USDT-BSC/USDC-BSC 的「`Start` 失败再 `Start_scan`」双路径：第一个成功后订单已是 PaySuccess，第二个因状态守卫返回 false，**天然防止双路径重复入账**。
- 修正 `Orders.ActualAmount` 注释：原写「保留4位小数」与实现（2位四舍五入）矛盾，已改为 2 位。

| 🟢 | 扫块时间窗严格过滤 | 临界到账（确认延迟）的真实付款被判过期漏单 | 是否加到账确认宽限期？宽限多久？ |

### 安全面（你已确认单实例、暂不处理，仅记录备查）
- `NotifyUrl` 无 SSRF 防护（商户传入地址服务端直接 POST）。
- JWT secret 用 `math/rand` 而非 `crypto/rand`，且进程重启随机生成。
- admin 写接口无 CSRF 防护、Cookie 未设 SameSite/secure。
- 改收款地址、SecretKey 等敏感操作无审计与告警。
- 下单金额无上限校验（极大值可能导致 float 溢出）。

> 以上安全项在「单实例 + 后台不暴露公网 / 仅可信管理员访问」前提下风险可控；若后台或下单接口暴露在公网且面向不可信方，建议逐项加固。

## 三、编译验证状态
**已通过。** 环境：Go 1.26.4，`E:\Go`（GOPATH=`E:\GoWork`）。

由于第三方库 `fvbock/endless` 使用 Unix 专属信号（`SIGUSR1/SIGUSR2/SIGTSTP`）无法在 Windows 本地编译，采用交叉编译验证（项目实际部署在 Linux）：
```
GOOS=linux GOARCH=amd64 go build ./...   # 退出码 0，全部包通过
GOOS=linux GOARCH=amd64 go vet ./db/sdb/ ./web/ ./cron/ ./mq/ ./BSC_USD/ ./USDC_BSC/   # 退出码 0
```
验证中修复了一个编译错误：`web/function.go` 下单循环里 `b.Next()` 的第二返回值类型是 `lbapi.Constrainable` 而非 `error`，原先 SetNX 复用了同名 `err` 变量导致类型冲突，已改用独立变量名 `setErr`。
